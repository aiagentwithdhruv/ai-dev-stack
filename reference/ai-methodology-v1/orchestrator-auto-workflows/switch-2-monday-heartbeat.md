# Switch 2 — Monday Heartbeat

> **Status:** PROVISIONAL — designed, not yet validated across 2+ projects.
> **Category:** Situational awareness
> **Cost:** ~5 minutes of orchestrator time every Monday
> **Benefit:** human starts the week with clarity instead of needing to reconstruct sprint state

---

## The Rule

Every Monday at 09:00 local time, the orchestrator produces a 5-bullet sprint-health report
and writes it to a file on the human's desktop.

- **v1:** manual. The orchestrator runs this on first contact Monday morning.
- **v1.1:** automated via cron or launchd (implement only after v1 is stable for 2+ weeks).

---

## Why

Without a heartbeat, the human reconstructs sprint state from memory at the start
of each week. That reconstruction is lossy — it skews toward recent events and
misses slow-moving risks. The heartbeat externalizes state so the human's first
question is "what do I act on today?" not "what happened last week?"

---

## What the report contains

The orchestrator reads:

1. `git log --since="last Monday" --oneline` — commits in the past week.
2. Last pytest run result (or equivalent test suite output).
3. New entries in `ERRORS.md` since last Monday.
4. Current task status across all modules (from `data/tasks.yaml` or the equivalent).
5. Any items explicitly flagged as "blocked" or "at risk."

Then produces exactly 5 bullets:

```markdown
# Sprint Health — {DATE}

1. **Commits:** {N} commits this week. Tip: {SHA} — "{SUBJECT}"
2. **Tests:** {PASS}/{TOTAL} passing. {N} new tests added. {FAILURES if any}
3. **Errors logged:** {N} new entries in ERRORS.md. Most critical: {SUMMARY}
4. **Module status:** {MODULE_A} {STATUS} · {MODULE_B} {STATUS} · {MODULE_C} {STATUS}
5. **Risks / blockers:** {ITEM or "None — clean week"}
```

The file is written to: `~/Desktop/SPRINT-HEALTH-{YYYY-MM-DD}.md`

---

## Template

```markdown
# Sprint Health — {YYYY-MM-DD}

Generated by orchestrator at 09:00. Covers {PREV_MONDAY} → {TODAY}.

1. **Commits:** {N} commits. Tip {SHA} — "{SUBJECT}". Active branch: {BRANCH}.
2. **Tests:** {PASS}/{TOTAL} passing ({FAIL} failing). New tests this week: {NEW}.
3. **Errors logged:** {N} new ERRORS.md entries.
   Most critical: {ERROR_TITLE} — {ONE_LINE_SUMMARY}.
4. **Module status:**
   - {MODULE_A}: {STATUS} ({DONE}/{TOTAL} tasks)
   - {MODULE_B}: {STATUS} ({DONE}/{TOTAL} tasks)
   - {MODULE_C}: {STATUS} — {NOTE_IF_ANY}
5. **Risks:** {BLOCKER_DESCRIPTION or "None identified."}

Next recommended action: {ONE_SENTENCE}
```

## v1.1 — Automated via launchd (macOS)

After v1 is stable for 2+ weeks, wire with a launchd plist:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
    "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.yourproject.heartbeat</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/bin/python3</string>
    <string>/absolute/path/to/heartbeat.py</string>
  </array>
  <key>StartCalendarInterval</key>
  <dict>
    <key>Weekday</key><integer>1</integer>
    <key>Hour</key><integer>9</integer>
    <key>Minute</key><integer>0</integer>
  </dict>
</dict>
</plist>
```

Do not automate until the manual version has run at least 4 times without complaint.

---

## Tradeoffs

- If the heartbeat is inaccurate, it creates false confidence. Quality over cadence.
- The 5-bullet constraint is intentional. If 5 bullets are not enough, the sprint has too much going on.
- Do not send the report via Slack, email, or any channel the human did not opt into.
  Desktop file is passive — the human finds it when ready.
